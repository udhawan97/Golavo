## The end-to-end test (`server/tests/test_phase6_e2e.py`)

One deterministic test, **no network / no live LLM**, composing the whole
pipeline and asserting byte-stable artifacts:

```
provenance-validate T0+T1 packs & the retained registry
  → ingest (France v Morocco flips scheduled→completed across refs)
  → evaluate the WC2026 fold (every candidate beats the baseline on log loss)
  → forward-SEAL on T0 (273c731492df, fixture still scheduled), before day-proxy kickoff
  → SCORE from strictly-newer T1 (the 2-0 result); the seal's bytes never change
  → aggregate the calibration record over the ledger (1 sealed, 1 scored)
  → build the evidence bundle (deterministic; carries the numeric whitelist)
  → run generate_narration() with a CANNED ADVERSARIAL response ("…win with
    probability 91%") → the whitelist rejects the whole narration → status
    local_only, narration None, "91%" never in the envelope
  → AI-off never contacts a model (tripwire transport, contacted == 0)
```

Seal, score, calibration, and the evidence bundle are each computed twice and
asserted byte-identical. Runs in ~14s; collected by `pytest -q` (CI `core` job
installs both packages).

## Exactly what is unsigned, and why

- **All desktop bundles are unsigned.** The user holds **none** of the signing
  secrets (`TAURI_SIGNING_PRIVATE_KEY(_PASSWORD)`, `APPLE_CERTIFICATE(_PASSWORD)`,
  `APPLE_SIGNING_IDENTITY`, `APPLE_ID`, `APPLE_PASSWORD`, `APPLE_TEAM_ID`,
  `MINISIGN_SECRET_KEY`). `build.sh` and `release.yml` skip each signing step when
  its secret is empty — no signed artifact is fabricated.
- macOS **Gatekeeper** / Windows **SmartScreen** warn on first launch (documented
  in `installation.md`: macOS right-click → Open; Windows More info → Run anyway).
- The **release is published as a GitHub pre-release** with an explicit
  unsigned-pre-release body; downloads are verifiable against `SHA256SUMS.txt`.
- The Tauri **updater is disabled** (`createUpdaterArtifacts:false`; feature off;
  `check_for_update` returns `disabled`) and its public key is still the
  placeholder — even a `--features updater` build is non-functional until a real
  key is supplied.
- **No SBOM** is wired; none is claimed.

## Remaining honest gaps (for a future signed 0.1.x / 0.2.0)

- **Desktop full-bundle build is CI-proven only at the tag build** (the Tauri
  bundle is never built in `ci.yml`). The shell launch + orphan watchdog are
  code-only (manual macOS verification per `codex-phase4.md`); no end-to-end
  desktop test exists. Windows bundle was never built locally.
- **Blocked / out of scope:** club lineups, injuries, xG, corners; goalscorer
  modeling; women's football; live club data + a club forward loop; confirmed-
  lineup forecasts; BYOK data-feed adapters; a hash-chained ledger (append-only
  JSONL only).
- **Signed/notarized release** pending the user's signing secrets. Fill
  `tauri.updater.conf.json` pubkey + the secrets, then the same `release.yml`
  publishes a full (non-pre) signed release.
- **The calibration record is empty** by design until a real upstream refresh
  seals a live fixture (one-command recipe unchanged from Phase 3).
