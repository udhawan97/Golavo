# Golavo v0.2 hardening — adversarial review

**Base SHA:** `30dd644` (v0.1.0 + 8 commits) · **Reviewer stance:** skeptical principal engineer + open-data lawyer + adversarial critic · **Method:** whole-system read, clean-checkout re-validation, empirical determinism probing, AI-guard exploitation attempts. Honesty over impressiveness — an empty dimension is a valid result.

Baseline at this SHA is green: `ruff` clean, core **148** passed, server **45** passed, provenance OK (7 packs), sample artifacts OK (8), license-isolation OK, UI build OK.

## Executive verdict

The product is **substantially what it claims to be**, and unusually honest about its limits. The deterministic guarantees that matter most hold under scrutiny: no chronological leakage in the seal path, machine-checked coherence between the 1X2 / expected-goals / exact-score matrix (all derived from one fitted joint distribution), an AI numeric whitelist that genuinely cannot emit an *unsupported* number, BYOK keys that never enter a prompt, and score-matrix determinism that survives ±128-ULP per-element perturbation. Most ✅/coverage/accuracy claims reproduce exactly.

The real exposure is **not** the model — it is the **trust boundary around already-sealed artifacts and the honesty of two security claims**. Three confirmed High findings: (H1) the read-only server serves on-disk artifacts with *zero* validation, so the "coherence is enforced on every load" and "immutable/auditable" promises do not hold at the API boundary; (H2) `SECURITY.md` and the packs docs assert **minisign signature-verified packs** with an "unsigned packs require override" control that **has no implementation**; (H3) the desktop sidecar's parent-death watcher is POSIX-only and both mis-fires `os.kill` and fails to detect orphaning on **Windows**, a shipped target — leaking a loopback server every session. None require blocked data or a contract change to fix.

## Dimension 1 — Model correctness & coherence

**No Critical/High findings. The coherence guarantee is real.** The sealed 1X2, expected goals, and exact-score matrix are all derived from one fitted joint matrix (`models/candidates.py:237-251`), and `score_matrix.assert_model_coherent` proves at seal time that the matrix marginals reproduce the sealed probs (tol `1e-5`) and its mean reproduces expected goals (tol `1e-4`), and that the stored grid+tail is a faithful re-bucketing (`score_matrix.py:235-272`). `assert_stored_coherent` re-checks from the JSON alone. The evidence-bundle scoreline ranking uses the same tie-break as `most_likely` (`evidence.py:105-118` vs `score_matrix.py:122-126`), so rank-1 == `most_likely`. No goalscorer allocation exists (correctly out of scope), so none can contradict the matrix.

**Leakage:** the seal excludes the target match by `match_id` and trains on `min(as_of, kickoff−1s)` (`artifacts.py:209-211`), which for a dates-only source excludes same-day fixtures; `assert_no_future_rows` guards every `fit` (`candidates.py:61,89,149`). No wall-clock is used anywhere in the seal/score/bundle paths (verified by grep). This is sound.

**Coherence is NOT enforced on the served path — see H1.** The one gap is that `validate_artifact` (which runs the coherence check) is only called on the *write* path and the CLI/CI validator, never when the API reads an artifact to serve it or to compute the calibration record. See H1.

## Dimension 2 — Data provenance & legality

**No active violation — but the barriers are convention-only.** At `30dd644` there is **no ODbL data in the tree**, no shipped code references it (`overlay-odbl` ships only a README), every vendored pack's declared files hash-match, and NOTICE is complete for the CC0 data actually shipped. The positive control that a source's license travels into every artifact (`artifacts.py` stamps `snapshot_descriptor`, which carries `license`) is genuine. **Findings M1–M3 are latent hardening gaps**, filed not fixed:

- **M1** — `scripts/check_license_isolation.sh` is two greps. It scans only `packs/core-cc0` (excluding `*.md`) and `core/` for the literal `import overlay_odbl`; it never scans `server/`, and it matches the *source name* (`odbl|openligadb`), not the data. Concrete: copying a Bundesliga results CSV into `packs/core-cc0/x.csv` trips nothing (the rows contain none of those tokens); a `pd.read_csv("packs/overlay-odbl/…")` in `server/` is invisible. The guard is a lint, not enforcement — which the code comments already imply, so this is a *strength-of-guarantee* gap, not an overclaim.
- **M2** — `validate_provenance.py` iterates only `manifest["files"]`; nothing asserts the directory contains *only* declared files, so an undeclared file would ship in a "CC0" pack unhashed and unaudited. (Verified: all seven packs are clean today.)
- **M3** — the *runtime* loader `ingest/snapshot.py` checks neither license nor a non-empty `files` list, so it is weaker than the CI validator at the exact point that joins pack data into the shared table.

Recommendation (all): a positive allowlist — CI asserts every file in a pack is manifest-declared and hash-listed; the runtime loader refuses a non-CC0 manifest into the core table; scan `server/` and forbid `packs/overlay-odbl/**` path-reads across `core/`+`server/`; re-run both guards inside `release.yml`.

## Dimension 3 — Reproducibility / determinism

**No findings beyond a coverage note (L7).** Regenerating the sample artifacts on macOS produced a **byte-identical** tree (`git status` clean) — the Poisson score-matrix path reproduces the committed baseline exactly. Empirically, the 9-dp grid rounding is far from fragile: across 120 fixture rates and all three goal families, **no** stored value flips under ±128-ULP *per-element* perturbation (a stand-in for cross-platform libm differences, which are typically ≤4 ULP). Canonicalization rounds probs to 6 dp with a drift correction on the largest outcome (`artifacts.py:73-83`), and ties break deterministically by dict insertion order. The residual (L7): byte-identity is not *asserted* across OSes in CI — the CI validator only checks each artifact's internal consistency. Recommend a determinism matrix job.

## Dimension 4 — Security (AI whitelist, injection, keys)

**No Critical/High findings — the AI guardrails are genuinely strong.** The served-narration gate (`ai/whitelist.unsupported_number_tokens`) admits a digit token only if it *exactly string-matches* the trusted `display` of a number that same claim references, binding value + unit + citation; spelled-out numbers and fractions are rejected outright; NFKC folds disguised unicode digits; betting lexicon and credential-shaped output hard-reject the whole narration (`narration.py:96-164`). I could not construct an **unsupported**-number leak on the served path.

**Prompt injection cannot change a displayed probability or exfiltrate a key.** Displayed probabilities come from the engine artifact, never the model; a successful injection can only degrade to `local_only`. Keys never enter the model: the `Transport` type is defined to never receive the key, and the real transports place it only in an HTTP header, never the request body/prompt (`ai_gateway.py:64-67,206-247`). Cloud providers cannot override `base_url` (`ai_gateway.py:110-111`) and Anthropic's URL is hardcoded, so a key-bearing request can't be redirected to an attacker host; local providers get no key and are loopback-validated (`ai_gateway.py:135-148`). Untrusted text is sanitized (control-token/fence stripping, length cap) and delimited as data (`ai/sanitize.py`).

**M5 (Med, known limitation, documented not fixed):** the whitelist guarantees "no *unsupported* number," not "every supported number in its correct semantic role." Because scoreline goal counts enter the whitelist as bare integers ("0","1","2") and equal-rounding percentages can share a `display`, a model (or a successful injection) could weave an allowed integer into a fabricated phrase, or attach a supported probability to the wrong outcome label, without tripping the numeric gate. This is inherent to numeric whitelisting without NLP; it is mitigated by AI being off by default and by the UI rendering engine-authored labels for number chips. Filed as a known limitation, not a defect.

## Dimension 5 — Honesty / claims audit

**The repo is unusually honest** — backtest numbers reproduce byte-for-byte from `build_model_cards.py`; "every candidate beats the climatological baseline on log loss on every fold" verifies across all leagues + internationals; AI-off-by-default, the empty calibration ledger, coverage counts (15/12/15/11/10 clean seasons), unsigned-desktop labeling, and the "planned" hash-chained ledger all check out. **One material overclaim (H2)** and three low doc drifts (L1, L2, and M4's wording):

- **H2 (High)** — `SECURITY.md:26` states data/model packs are "**Signature-verified (minisign) against a pinned public key** and hash-checked before load. **Unsigned packs require explicit user override.**" as an *active* commitment (unlike the Updates row, which is hedged "planned"). No minisign verification, no pinned public key, and no override path exist anywhere; only per-file sha256 self-hashing runs, and the manifest lives *inside* the unsigned pack, so the control is corruption-detection, not authenticity. `packs/README.md:3-5` ("versioned, **signed** packs … minisign-signed … before the app loads it") and `README.md:157` ("**Signed packs** … wired") repeat it at lower severity. Fixed by rewording to the true mechanism and marking signature-verification "planned (ADR-0001), not implemented."
- **L1** — `SECURITY.md:27` "Phase 0 has no updater" is stale; the (gated) Tauri updater has landed. Corrected.
- **L2** — `docs-site/package.json` version `0.0.0`. Corrected in the version bump.

## Dimension 6 — UI/accessibility, desktop lifecycle, release integrity

**Heatmap accessibility is well built — no findings.** `ScoreMatrixHeatmap.tsx` prints each cell's percentage as text (color is redundant, satisfying WCAG 1.4.1), uses real table semantics (`<caption>`, `scope` headers, per-cell `aria-label`), flags the most-likely cell three non-color ways (★ glyph, outline, aria text), bounds contrast to AA, and respects `prefers-reduced-motion`. **Unsigned labeling is exemplary** — every user-facing surface states the unsigned reality.

- **H3 (High)** — desktop sidecar orphan on Windows (a shipped target). `server/golavo_server/sidecar.py:_watch_parent`/`_pid_alive` use POSIX semantics: reparent-to-init detection (`os.getppid()`) never fires on Windows, and `os.kill(pid, 0)` on Windows is **not** a probe — CPython routes signal 0 (`CTRL_C_EVENT`) through `GenerateConsoleCtrlEvent`/`TerminateProcess` and raises `OSError` (not the caught `ProcessLookupError`), killing the watcher thread. Concrete: quitting Golavo on Windows kills the PyInstaller bootloader but orphans the reparented Python/uvicorn child, whose watcher can neither detect the orphaning nor safely probe → the sidecar leaks on `127.0.0.1` holding its port/token, accumulating across launches. Fixed by platform-dispatching the watcher (Windows: a non-killing `OpenProcess`+`WaitForSingleObject` liveness probe; broadened `except OSError`), POSIX path unchanged.
- **M4 (Med)** — `.github/workflows/release.yml:117-119` "Sign checksums (minisign, if key configured)" runs `echo …` — it signs nothing even with the secret set — while the release body (`:136-137`) tells users to "Verify every download against `SHA256SUMS.txt`", an unauthenticated file on the same page; and a set `MINISIGN_SECRET_KEY` flips the release to a full (trusted-looking) release (`:127`) while still signing nothing. Wording corrected under H2; a real minisign implementation is deferred.
- **M6 (Low–Med)** — Casual mode presents the same sealed probabilities as Expert but omits the distribution/tail and the **calibration/ledger pointer** ("go check how these have matched reality"), so the two modes hedge the same number differently. `ForecastDetail.tsx`. Filed.
- **L3–L5** (updater pubkey placeholder; 90 s splash on boot-failure; PID-reuse race) filed.
