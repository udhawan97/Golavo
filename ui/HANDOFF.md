## What this is

A beautiful, honest, **read-only** UI over sealed forecast artifacts, driven entirely
by mock JSON conforming to frozen contract **v0.1.0**. No live integrations, no invented
endpoints, no backend changes. Not a betting product — no odds framing anywhere.

`src/lib/api.ts` chooses live vs mock from `import.meta.env.VITE_GOLAVO_API` and validates

or a `{forecasts:[…]}` envelope). No other endpoints are assumed.

## Accessibility verification

- **Contrast (WCAG 2.2 AA):** measured with a luminance script in-page. Body text 15.2:1;
  muted 7.35:1; dim 4.7:1. Status chips (small bold) pass in both themes — dark 4.77–7.02:1,
  light 4.98–5.83:1 (light-theme chip text was darkened to clear 4.5:1).
- **Keyboard / focus:** all interactive elements are native (`a`, `button`, `select`); a
  visible 3px gold focus ring (`:focus-visible`) is applied; a skip-link jumps to `#main`.
- **Structure:** single `h1` per view, no heading-level skips; `header`/`nav`/`main`/`footer`
  landmarks; tables use scoped `th`; the reliability `svg` has an `aria-label` summary;
  the probability bar exposes an accessible text summary and hides its decorative segments.
- **Reduced motion:** a global `prefers-reduced-motion: reduce` rule neutralises transitions
  and the skeleton shimmer; the brand lockup SVG carries its own reduced-motion fallback. (The
  handoff screenshots were captured with reduced-motion emulated.)
- **Responsive:** no horizontal page scroll at 375 px on any view; the wide 5-column eval table
  scrolls inside its own container.

## Contract gaps — notes for Codex (NOT changed here)

Per the brief, I did not touch the canonical contract. These are shapes a view wanted that
v0.1.0 cannot currently express; each was worked around read-only without inventing fields:

1. **No machine-readable void reason.** `status: "voided"` has no counterpart to abstained's
   `abstain_reason` (nor a `voided_at_utc`). The UI shows a generic "excluded from scoring"
   banner. Consider an optional `void_reason: string | null` (+ `voided_at_utc`).
2. **Supersession is backward-only.** An artifact carries `supersedes` (points at the older
   seal) but nothing points forward. The UI derives "superseded by …" by scanning the loaded
   list — fine while `/forecasts` returns whole chains, but it would break under pagination or
   filtering. Consider an optional `superseded_by` (or a documented guarantee that chains are
   returned intact).
3. **`EvalSummary` models lack a `family`.** `FoldModel` has only `model_id`; the UI prettifies
   known id prefixes (`clim`/`elo`/…) for display. An explicit `family` or `display_name` would
   be more robust.
4. **List endpoint shape is unspecified.** The brief names `GET /api/v1/forecasts` but not
   whether it returns a bare array or an envelope. The UI accepts both; worth pinning in the
   contract.
5. **No "kickoff passed but unscored" signal.** A superseded seal stays `status: "sealed"` with
   a past kickoff. The UI infers "awaiting kickoff" from `kickoff_utc` vs now to avoid mislabeling
   it. Not blocking; noting for completeness.

## Match Search & the per-match Commentator's Notebook (later addition)

A read-only search and filter-browse surface over the committed 100k-match index, plus a Commentator's
Notebook for *any* searched match. Additive to the workbench above — the sealed forecast
views, the Ledger, and the AI panel are unchanged. Full engineering note:
[`docs/handoff/match-search.md`](../docs/handoff/match-search.md).

### Views → data source

| View | Route | Endpoint (`VITE_GOLAVO_API` set) | Mock fallback (default) |
| --- | --- | --- | --- |
| Match search | `#/matches` | `GET {base}/api/v1/matches/search?q=&competition=&status=&limit=&offset=` + `GET {base}/api/v1/matches/competitions` | `src/mocks/matches.json` |
| Match detail + Commentator's Notebook | `#/match/{id}` | `GET {base}/api/v1/matches/{id}` + `GET {base}/api/v1/matches/{id}/notebook` | same mock, by id |

`src/lib/api.ts` gains `searchMatches`, `fetchCompetitions`, `fetchMatch`, and
`fetchMatchNotebook`. Live mode hits the endpoints; a `503` (index unavailable) and mock
mode both fall back to `src/mocks/matches.json` — a small **synthetic** contract fixture,
labelled as such (not real fixtures). Contract types are additive in `src/lib/contract.ts`:
`MatchRow`, `MatchSearchResponse`, `MatchDetailResponse` (with
`linked_by: "match_id" | "fixture" | null`), `CompetitionsResponse`, and
`MatchNotebookResponse` (reusing the Phase-7 `CommentatorsNotebook` type + `NotebookFacts`
renderer). The search box is debounced; status and competition filters can browse without query
text; Load-more pages by `offset`, keyed so a stale page never appends to a newer query.

### Honest states (rendered from engine fields alone)

**Search result badge** — classifies by `is_complete` first, never `kickoff` vs now:

| Condition | Badge |
| --- | --- |
| `is_complete` | `Played H–A` |
| not complete, `kickoff` in the future | `Upcoming` |
| not complete, `kickoff` in the past | `Result not in snapshot` (a midnight-UTC day-proxy kickoff is not proof of a played match) |
| a ledger forecast links to the row | `Sealed forecast` chip alongside |

**Match detail forecast block** — four states:

| State | Shown |
| --- | --- |
| Sealed forecast linked | link to the sealed `#/forecast/{id}` (with `linked_by` provenance) |
| Played, no seal | the recorded final score + "Golavo never retro-forecasts a played match" |
| Future, no seal | "Sealing from inside the app lands in a future release; today, seals are written by the engine CLI" — **club matches** additionally note forward sealing covers internationals (historical-backtest-only) |
| Past kickoff, no recorded score, no seal | "No result on record" (not "upcoming") |

The **Commentator's Notebook** block is always shown for a found match, fed by the per-match
notebook endpoint and rendered through the shared `NotebookFacts` renderer. It is
descriptive history, subordinate to any seal, and never a forecast.

### Deferred / notes

- **Seal-from-UI is deferred — the API stays all-GET.** No write path was added here.
- Match ↔ forecast linking is served from the **real ledger only**, so synthetic sample
  ids can never attach to a real fixture (see the handoff note's sample-mode invariant).
- `status=upcoming` is empty until a pack refresh adds forward-dated fixtures; Load-more and
  the past-incomplete state are code-verified (server tests + the code path), not visibly
  reproducible from the small mock alone.
